# Propagate CLI Scenarios

These workflows use placeholders such as `<handle>`, `<scope>`, and `<sha>`. Do not replace placeholders in docs with real env values.

When command usage is unclear, run `propagate --help`, `propagate <group> --help`, or `propagate <group> <command> --help`.

## Inspect A Repository

Use read-only status commands before changing anything:

```bash
propagate config status --json --non-interactive
propagate team status --json --non-interactive
propagate env status --scope dev --json --non-interactive
```

If the cloud cannot be reached, pass `--api-url VALUE` or configure it outside docs and prompts.

## Initialize A New Team Project

Preview setup first:

```bash
propagate init --handle <handle> --team-name <team-name> --skip-agent-guidance --dry-run --json --non-interactive
```

After reviewing the env file mappings and getting approval:

```bash
propagate init --handle <handle> --team-name <team-name> --skip-agent-guidance --yes --non-interactive
```

Use `--agent-guidance` instead of `--skip-agent-guidance` when the human wants `AGENTS.md` updated.

## Request Team Access

**Git-mediated join:** the command adds a pending request to `propagate.yaml` for review in Git.

When **PIN invite codes** exist for the team (planned), interactive `propagate team join` first asks whether to **Request to join** (git-mediated only) or **Join by invite code** (PIN flow). If there are **no** active invite codes, the CLI proceeds with the git-mediated join without that prompt. Non-interactive usage will need explicit join-mode flags once invites ship.

Preview the join request:

```bash
propagate team join --handle <handle> --role developers --scope dev=read --dry-run --json --non-interactive
```

Write the Git-reviewed request:

```bash
propagate team join --handle <handle> --role developers --scope dev=read --non-interactive
```

Commit the resulting `propagate.yaml` change and ask an admin to review it before `propagate config push`.

## Approve Or Decline Pending Joins

Inspect the pending state:

```bash
propagate team status --json --non-interactive
propagate config status --json --non-interactive
```

Preview decisions:

```bash
propagate config push --approve-join <sha> --skip-join <sha> --dry-run --json --non-interactive
```

After human approval, publish:

```bash
propagate config push --approve-join <sha> --skip-join <sha> --yes --non-interactive
```

Every pending join needs an explicit `--approve-join`, `--decline-join`, or `--skip-join` decision in non-interactive mode.

## Pull Cloud Config

Preview the cloud-to-local config update:

```bash
propagate config pull --dry-run --json --non-interactive
```

If the preview is approved and local overwrite risk is understood:

```bash
propagate config pull --yes --non-interactive
```

Review and commit the resulting `propagate.yaml` revision change.

## Create A Scope

Preview local metadata:

```bash
propagate scope create staging --env-file .env.staging --dry-run --json --non-interactive
```

Write the local scope metadata:

```bash
propagate scope create staging --env-file .env.staging --non-interactive
```

Use `propagate config edit` after scope creation to add env file mappings or move declaration metadata into the new scope. Scope creation does not copy metadata or values from existing scopes.

The scope becomes cloud-visible only after an admin reviews and runs `propagate config push`.

After publication, seed the mapped target env file locally and use `propagate env push --scope staging --dry-run` to preview the encrypted upload. Keep env values out of chat, docs, tests, and pull request text.

## Edit Config Metadata

`propagate config edit` is interactive. Use it when a human is present:

```bash
propagate config edit
```

Agents should not drive this editor blindly. For automation, propose the intended metadata change and let the human decide whether to run the interactive editor or make a reviewable metadata-only edit.

## Run With Injected Env Values (Recommended)

The preferred way to use secrets locally. No `.env` files written to disk — secrets exist only in process memory:

```bash
propagate run --scope dev -- npm run dev
```

`propagate run` decrypts values locally and passes them only to the child process environment. It does not write `.env` files, and injected cloud values override inherited environment variables with the same names for that child process.

Do not use `propagate run` to inspect or print secrets. Child process output is not sanitized by Propagate, so avoid commands such as `env`, `printenv`, or debugging tools that dump environment variables.

For prod process injection, get explicit human approval first:

```bash
propagate run --scope prod --yes -- ./bin/maintenance-task
```

If `propagate run` reports duplicate variable names across env file mappings, use `propagate env pull` for file-based workflows or resolve the duplicate metadata before retrying.

## Pull Env Values (File-Based Fallback)

Use `env pull` when your tooling requires a `.env` file on disk (e.g., Docker Compose, certain frameworks):

```bash
propagate env pull --scope dev --dry-run --json --non-interactive
```

After human approval for file creation, overwrites, or prod writes:

```bash
propagate env pull --scope dev --yes --non-interactive
```

Do not copy values from local env files into chat, docs, tests, fixtures, or commits.

## Push Local Env File Changes

Preview the encrypted upload plan:

```bash
propagate env push --scope dev --dry-run --json --non-interactive
```

After human approval of additions, changes, and removals:

```bash
propagate env push --scope dev --yes --non-interactive
```

If the command reports a secret version conflict, run `propagate env pull --scope dev --dry-run` and review before retrying.

## Set One Env Value

Interactive mode prompts without echo:

```bash
propagate env set NAME --scope dev
```

If `--scope` is omitted and more than one scope exists, the command asks the user to choose the target scope before prompting for the value. If exactly one scope exists, that scope is used automatically.

Non-interactive mode requires an approved secure stdin channel and explicit confirmation:

```bash
propagate env set NAME --scope dev --value-stdin --yes --non-interactive
```

Keep `--scope` explicit in non-interactive runs whenever the repository has multiple scopes.

Never pass the value as a positional argument. Never show the value in terminal summaries, docs, issue text, pull requests, tests, or agent messages.

## Handle Permission Or Revision Errors

When a command reports permission denied, pending join required, stale config revision, or missing scope access, stop and report the blocker. The usual safe next commands are:

```bash
propagate team status --json --non-interactive
propagate config status --json --non-interactive
propagate config pull --dry-run --json --non-interactive
```

Do not retry write-capable commands with broader flags unless the human explicitly approves the specific next action.
